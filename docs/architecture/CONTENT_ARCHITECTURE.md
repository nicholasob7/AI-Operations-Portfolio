# Content Architecture

## Purpose

This document defines how the website’s canonical selected public truth source should be structured and projected to the three first-class consumers.

It operates under the authority of `PUBLICATION_BOUNDARY.md` and `FIRST_CLASS_CONSUMERS.md`.

The site still pursues one canonical selected public truth source. It does not construct that source through maximal evidentiary claim, proof-density ranking, hard deduplication, suspicious treatment of omission, or audit-style surface ranking.

The correct method is relationship-first hydration, followed by pruning.

## Canonical Selected Public Truth Source

The canonical source is the selected public truth Nicholas intentionally publishes for this bounded resume/portfolio website.

It should include the public claims, relationships, capability signals, confidence signals, judgment signals, direction signals, and discretion signals that are appropriate for the site.

It is not exhaustive.

It is not a hidden evidence vault.

It is not a demand for corporate-sensitive, client-sensitive, or operational substrate.

All intentionally published website content remains primary public truth in its presentation mode. Summary content is not lesser truth because it is summary. Quality and personal content are not lesser truth because they are less operationally detailed.

## Hydration From Bounded Public Claims

Content architecture should begin with intentionally published bounded public claims.

Hydration means carrying forward the useful relationships those claims express or reasonably support.

For example, a public project description may connect:

- role context
- skill
- action
- testing
- iteration
- judgment
- outcome
- confidence
- direction
- professional discretion

Hydration must preserve relationship-bearing signal before pruning.

Hydration must not become invention. It must not fabricate concealed specifics, app names, clients, scripts, logs, timestamps, credential paths, deployment internals, or private operational detail.

## Relationship-Bearing Architecture

The source should make important relationships legible without turning the site into metadata bureaucracy.

Relevant relationships may include:

- skill to project
- action to outcome
- testing to confidence
- judgment to boundary-setting
- role context to plausible capability inference
- language quality to semantic control
- governance direction to technical identity
- public sanitization to professional discretion
- project status to claim strength

Navigation, grouping, sequencing, and labels may assist hydration. They should not silently become heavy truth claims by layout alone.

Weak explicit signal is not weak competence. Absence of a named tool is not absence of capability. Relationship-rich bounded public descriptions may support credible capability inference without disclosure of concealed specifics.

## Pruning Discipline

After hydration, prune the source back to what is justified, bounded, recruiter-useful, and appropriate for public release.

Prune:

- concealed specifics
- unsupported speculation
- inflated claims
- corporate-sensitive or client-sensitive material
- operational substrate not intentionally public
- detail that adds disclosure burden without improving public professional signal

Retain:

- bounded public claims
- useful summaries
- relationship-bearing capability signal
- confidence supported by action, testing, iteration, deployment, or review
- judgment shown through boundaries, scope control, and separation of confirmed outcomes from open questions
- direction shown through project choice, language, governance, and technical identity
- discretion shown through appropriate sanitization and non-disclosure

Pruning must not become collapse. Do not reduce relationship-rich public content into evidentiary nullity or bare fact shards.

## Projection To First-Class Consumers

The same canonical selected public truth source may produce different projections or services for the three first-class consumers.

Machine consumer:

- should recover intentionally public claims efficiently
- should recover relationship-bearing signal where possible
- should preserve reasonable logical conclusions from public relationships
- should distinguish withheld detail from absent capability
- should not rely unnecessarily on human-style interaction state
- must not fabricate concealed specifics

Human reader:

- should receive paced, selective, confidence-forming presentation
- should be able to form a credible first-fit judgment quickly
- should receive readable prose, hierarchy, and navigation
- should not be burdened with archive logic, schema-heavy exposition, or exhaustive proof framing

PDF consumer:

- should receive a stable public projection suitable for sharing, review, and printing
- may receive denser or more fixed presentation
- should remain aligned enough with selected public truth
- must not become a competing truth source

Hidden-by-UX is acceptable. Hidden-from-retrieval of intentionally public content is the real machine-consumer concern.

## Alignment And Drift

Different projections do not need to be identical.

They may differ in:

- density
- sequencing
- wording
- layout
- interaction
- level of explicit relationship labeling
- update cadence

They should not differ by maintaining separate public truth regimes.

Project-level machine projections must not create parallel slice/detail entries for the same artifact unless the entries serve distinct consumer roles not already served by another surface.

PDFs may lag if separately authored. That is a projection-management issue, not a separate truth authority.

When public website content, machine-readable content, and PDFs diverge materially, the issue is alignment and drift, not evidence failure.

## Failure Modes To Reject

Reject these failure modes:

- returning to maximal evidentiary claim as the construction method
- treating omission as a hole to be filled
- ranking truth by proof density
- treating project-detail pages as more true because they are closer to operational substrate
- demoting summary, quality, personal, or resume content into lesser truth status
- replacing relationship-bearing meaning with fact-shard bureaucracy
- collapsing sanitized public signal into weak or null capability
- fabricating concealed specifics for machine projection
- treating PDFs as a separate truth regime
- demanding identical wording across projections
- allowing layout, grouping, or navigation to imply unsupported heavy claims
